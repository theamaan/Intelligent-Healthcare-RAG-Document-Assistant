# Intelligent Healthcare RAG Document Assistant

![CI](https://github.com/theamaan/Intelligent-Healthcare-RAG-Document-Assistant/actions/workflows/ci.yml/badge.svg)
![Python](https://img.shields.io/badge/python-3.11%2B-blue)
![License](https://img.shields.io/badge/license-MIT-green)

A fully **local**, **open-source** retrieval-augmented generation (RAG) system for healthcare documents.  
Upload insurance policies, care guidelines, or medical records and ask contextual questions.  
Answers are grounded in the document with **page-level citations** — no hallucinations, no cloud, no API keys.

---

## Architecture

```
┌─────────────────────┐       HTTP        ┌──────────────────────────────────┐
│   Streamlit UI      │ ───────────────▶  │  FastAPI Backend                 │
│   localhost:8501    │                   │  localhost:8000                  │
└─────────────────────┘                   │                                  │
                                          │  POST /upload                    │
                                          │    → PyMuPDF / python-docx       │
                                          │    → RecursiveCharacterSplitter  │
                                          │    → sentence-transformers       │
                                          │    → FAISS (saved to disk)       │
                                          │                                  │
                                          │  POST /query                     │
                                          │    → FAISS MMR retrieval         │
                                          │    → Ollama LLM (local)          │
                                          │    → grounded answer + citations │
                                          │                                  │
                                          │  GET  /history/{doc_id}          │
                                          │  GET  /eval/latest               │
                                          │  GET  /health                    │
                                          └──────────────┬───────────────────┘
                                                         │
                                          ┌──────────────▼───────────────────┐
                                          │  Ollama  localhost:11434         │
                                          │  llama3.2:3b  (default)          │
                                          │  phi3:mini    (fallback)         │
                                          └──────────────────────────────────┘
```

**Key design decisions:**
- Citations are deterministic metadata joins (source file + page number + chunk index), not generated by the LLM.
- FAISS indexes are persisted per document under `data/vector_stores/<doc_id>/`.
- All data stays on your machine. Nothing is sent to external services.

## Quick Start (Local — no Docker)

```bash
# 1. Clone the repo
git clone https://github.com/theamaan/Intelligent-Healthcare-RAG-Document-Assistant.git
cd Intelligent-Healthcare-RAG-Document-Assistant

# 2. Create virtual environment and install dependencies
python -m venv .venv
.venv\Scripts\activate          # Windows PowerShell
# source .venv/bin/activate     # macOS / Linux

pip install -r requirements.txt

# 3. Create your .env (copy the example; defaults work out of the box)
copy .env.example .env          # Windows
# cp .env.example .env          # macOS / Linux

# 4. Start Ollama and pull the default model (~2 GB download, first time only)
ollama serve
ollama pull llama3.2:3b

# 5. Start the backend (new terminal)
.venv\Scripts\uvicorn backend.app.main:app --reload --host 0.0.0.0 --port 8000

# 6. Start the frontend (new terminal)
.venv\Scripts\streamlit run frontend/app.py

# 7. Open http://localhost:8501 in your browser
```

**Windows PowerShell helpers:**
```powershell
.\scripts\run_backend.ps1
.\scripts\run_frontend.ps1
```

**Makefile helpers (Git Bash / WSL / macOS / Linux):**
```bash
make setup     # create venv + install deps
make backend   # start uvicorn
make frontend  # start streamlit
make test      # run pytest
make eval      # run RAGAS evaluation
```

## Usage Workflow

1. **Upload a document** — drop a PDF or DOCX (≤ 15 MB) via the sidebar
2. **Index** — the backend extracts text, chunks it, embeds it, and saves a FAISS index
3. **Ask questions** — type a question in natural language; the system retrieves relevant chunks, generates a grounded answer, and returns page-level citations
4. **Review citations** — each citation shows the source file, page number, chunk index, and a text excerpt

### Example questions for an insurance policy document
```
Is hospitalization covered under this plan?
What is the annual deductible for in-network services?
Are mental health services covered?
What is excluded from coverage?
How do I request prior authorization for a specialist visit?
What happens when I reach my out-of-pocket maximum?
```

## API Reference

### `POST /upload`
Upload and index a PDF or DOCX file.

```bash
curl -X POST http://localhost:8000/upload \
  -F "file=@your_policy.pdf"
```

**Response:**
```json
{
  "doc_id": "doc_a3f1b2c9d4",
  "filename": "your_policy.pdf",
  "chunk_count": 142,
  "message": "Document uploaded and indexed successfully"
}
```

---

### `POST /query`
Ask a question against an indexed document.

```bash
curl -X POST http://localhost:8000/query \
  -H "Content-Type: application/json" \
  -d '{"doc_id": "doc_a3f1b2c9d4", "question": "Is hospitalization covered?"}'
```

**Response:**
```json
{
  "doc_id": "doc_a3f1b2c9d4",
  "question": "Is hospitalization covered?",
  "answer": "Yes, hospitalization is covered under inpatient benefits subject to the policy terms.",
  "citations": [
    {
      "source": "your_policy.pdf",
      "page_number": 12,
      "chunk_index": 47,
      "excerpt": "Inpatient hospital services are covered when medically necessary..."
    }
  ],
  "confidence": 0.8
}
```

> **Confidence** is a proxy score (0–1) derived from the number of retrieved chunks that supported the answer. It is not a model probability.

## Offline RAG Evaluation (RAGAS)

RAGAS measures answer faithfulness, context precision/recall, and answer relevancy using the local LLM as a judge.

**Step 1 — Upload your document and note the `doc_id`:**
```bash
curl -X POST http://localhost:8000/upload -F "file=@your_policy.pdf"
# returns: {"doc_id": "doc_a3f1b2c9d4", ...}
```

**Step 2 — Edit `data/test_sets/sample_qa.json`:**
Replace every `"replace_with_real_doc_id"` with your actual `doc_id`.  
Adjust `ground_truth` values to match content in your specific document.

**Step 3 — Run the evaluation:**
```bash
python -m backend.app.evaluation.ragas_eval
# Evaluation written to: data/eval_results/eval_20260407_120000.json
```

**Step 4 — View results** via the Evaluation tab in the Streamlit UI, or via the API:
```bash
curl http://localhost:8000/eval/latest
```

## Configuration

All settings are controlled via environment variables (see `.env.example`).

| Variable | Default | Description |
|---|---|---|
| `OLLAMA_BASE_URL` | `http://localhost:11434` | Ollama server URL |
| `OLLAMA_MODEL` | `llama3.2:3b` | Model used for generation |
| `EMBEDDING_MODEL` | `sentence-transformers/all-MiniLM-L6-v2` | Embedding model (downloaded on first use) |
| `RETRIEVAL_K` | `5` | Number of chunks returned by MMR retriever |
| `RETRIEVAL_FETCH_K` | `20` | Candidate pool size for MMR |
| `MAX_FILE_SIZE_MB` | `15` | Maximum upload file size |

**Lower-memory alternative model:**
```env
OLLAMA_MODEL=phi3:mini
```
```bash
ollama pull phi3:mini
```

## Troubleshooting

### Ollama not connecting
- Ensure Ollama is running: `ollama serve`
- Verify the model is downloaded: `ollama list` — should show `llama3.2:3b`
- Check `GET /health` endpoint: `curl http://localhost:8000/health`

### "No readable text found in document"
- The PDF may be image-only (scanned). PyMuPDF extracts text layers only.
- Try a text-based PDF or convert using OCR tools before uploading.

### Slow responses
- First query after startup is slower due to embedding model loading (cached after first call).
- For CPU-only machines, generation may take 30–90 seconds depending on the model.
- Use `phi3:mini` for faster but slightly lower quality responses.

### Docker: Ollama health check keeps failing
- The `ollama/ollama` image may take 30+ seconds to start on the first run.
- Run `docker compose logs ollama` to check its startup state.
- If stuck, pull the model manually: `docker compose exec ollama ollama pull llama3.2:3b`

### Evaluation fails with placeholder doc_id error
- Open `data/test_sets/sample_qa.json` and replace all `"replace_with_real_doc_id"` values with real `doc_id`s from `POST /upload`.

### Port conflicts
- Change ports in `.env` and `docker-compose.yml` if `8000` or `8501` are in use.
